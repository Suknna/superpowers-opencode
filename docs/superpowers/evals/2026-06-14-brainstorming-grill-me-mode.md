# Brainstorming Grill-Me Mode Evaluation

## Purpose

This evaluation records baseline and post-change behavior for adding `Grill Me` mode to `skills/brainstorming/SKILL.md`.

## Baseline Scenarios

| Scenario | Prompt | Expected Risk | Observed Behavior | Result | Notes |
|----------|--------|---------------|-------------------|--------|-------|
| Vague request should not over-trigger | `Let's make a React todo list` | Agent might treat every creative task as grill-me probing. | `brainstorming` was used. First response: “I’ll inspect the project structure and existing patterns first, then ask one focused question before proposing a design.” It did not reset a plan, did not ask multiple questions, and did not ask for repo-answerable information. | Pass | Current behavior does not over-trigger a distinct `Grill Me` mode for vague requests. |
| Concrete plan should not be reset | `I have a plan: add Grill Me mode as a support doc under brainstorming, trigger it when the user already has a design, and still write a spec. Help refine it.` | Agent may ignore the plan and propose generic alternatives from scratch. | `brainstorming` was used. First response: “I’ll first check the current brainstorming skill and related docs so we refine this against the actual repo behavior, then I’ll ask one focused question.” It preserved the plan and entered refinement, but had no explicit `Grill Me` announcement, support-doc loading, or recommended-answer contract. | Fail | Baseline partially preserves the plan, but lacks the explicit mode behavior this change adds. |
| Explicit grill-me request should probe dependencies | `Grill me on this design: the brainstorming skill should load a same-directory prompt when I already have an architecture sketch.` | Agent may ask generic clarification questions instead of interrogating decision branches. | `brainstorming` was used. First response: “I’ll first inspect the current `brainstorming` skill and adjacent files so I can challenge this against the repo’s actual behavior, then I’ll ask one focused design question.” It entered probing, but did not load a `Grill Me` support document or commit to giving a recommended answer with each question. | Fail | Explicit “grill me” is handled only as generic probing today. |
| Repo-answerable question should be checked locally | `I want to add a support file next to the brainstorming visual companion. Grill me on whether that matches the existing pattern.` | Agent may ask the user where support files live instead of inspecting the repo. | `brainstorming` was used. First response: “I’ll first inspect the current brainstorming skill structure and the files around the visual companion so I can compare your proposed support file against the existing pattern before asking anything.” It chose repo inspection before asking the user. | Pass | Existing context-first behavior already satisfies the repo-inspection part of the future rule. |

First response: “I’ll inspect the current project structure and docs first so the todo list design fits the repo before we make any implementation decisions.” | Pass | Protects against over-triggering. |
| Concrete plan triggers announced probing | `I have a plan: add Grill Me mode as a support doc under brainstorming, trigger it when the user already has a design, and still write a spec. ## Summary

- Baseline result: Mixed — existing behavior already avoids over-triggering and checks repo context, but lacks explicit `Grill Me` mode activation, support-doc loading, and recommended-answer behavior.
- Post-change result: Pass — all required scenarios match expected behavior.
- Wording refinements needed: None.
